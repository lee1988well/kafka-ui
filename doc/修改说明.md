# kafka-ui 定制化修改说明

本文档记录了对 kafka-ui 官方源码所做的全部修改，修改目标为修复实际使用中发现的五个问题。

---

## 一、RBAC 通配符集群支持

**问题**：通过界面动态添加新 Kafka 集群后，RBAC 配置中没有对应条目，导致普通用户无法看到新集群。

**解决方案**：在 RBAC 配置中支持使用通配符 `"*"` 匹配所有集群，代码后端和前端均需适配。

### 修改文件一：后端权限校验

**文件**：`kafka-ui-api/src/main/java/com/provectus/kafka/ui/service/rbac/AccessControlService.java`

在 `filterCluster` 方法中增加通配符匹配逻辑：

```java
private Predicate<Role> filterCluster(String cluster) {
  return grantedRole -> grantedRole.getClusters()
      .stream()
      .anyMatch(c -> "*".equals(c) || cluster.equalsIgnoreCase(c));
}
```

### 修改文件二：前端权限校验

**文件**：`kafka-ui-react-app/src/lib/permissions.ts`

在 `isPermitted` 和 `isPermittedToCreate` 两个函数中，集群查找时增加通配符回退：

```ts
const clusterMap = roles.get(clusterName) || roles.get('*');
```

**使用方式**：在 RBAC 配置中，将 `clusters` 字段设置为 `["*"]` 即可匹配所有集群。注意 YAML 中通配符必须加引号：

```yaml
roles:
  - name: readonly
    clusters:
      - "*"   # 必须加引号，否则 YAML 解析报错
    subjects:
      - ...
    permissions:
      - ...
```

## 二、Topic 消息查看默认行为优化

**问题**：
- 默认每页显示 100 条消息，数量过多，加载慢
- 默认按最旧排序（Oldest First），通常更希望看最新消息
- 切换筛选条件后，每页数量仍然是 100 条

### 修改文件一：全局默认每页数量

**文件**：`kafka-ui-react-app/src/lib/constants.ts`

```ts
// 修改前
export const MESSAGES_PER_PAGE = '100';

// 修改后
export const MESSAGES_PER_PAGE = '10';
```

### 修改文件二：筛选提交时的每页数量

**文件**：`kafka-ui-react-app/src/components/Topics/Topic/Messages/Filters/Filters.tsx`

```ts
// 修改前
const PER_PAGE = 100;

// 修改后
const PER_PAGE = 10;
```

同时将 `seekType` 的设置移到 `isSeekTypeControlVisible` 条件块外部，确保无论是否选择了分区，`seekType` 都能根据 `seekDirection` 正确设置：

```ts
switch (seekDirection) {
  case SeekDirection.FORWARD:
    props.seekType = SeekType.BEGINNING;
    break;
  case SeekDirection.BACKWARD:
  case SeekDirection.TAILING:
    props.seekType = SeekType.LATEST;
    break;
  default:
    props.seekType = currentSeekType;
}

if (isSeekTypeControlVisible) {
  // 仅在用户手动指定 Offset 或 Timestamp 时才覆盖 seekType
  if (offset && currentSeekType === SeekType.OFFSET) {
    props.seekType = SeekType.OFFSET;
  }
  if (timestamp && currentSeekType === SeekType.TIMESTAMP) {
    props.seekType = SeekType.TIMESTAMP;
  }
  // ...seekTo 逻辑不变
}
```

### 修改文件三：默认排序方向与初始 URL 参数

**文件**：`kafka-ui-react-app/src/components/Topics/Topic/Messages/Messages.tsx`

1. 将默认 seek 方向改为 `BACKWARD`（Newest First）：

```ts
// 修改前
const defaultSeekValue = SeekDirectionOptionsObj[SeekDirection.FORWARD];

// 修改后
const defaultSeekValue = SeekDirectionOptionsObj[SeekDirection.BACKWARD];
```

2. 在初始化 URL 参数时，同步写入 `seekDirection` 和 `seekType` 默认值，避免与 `Filters.tsx` 的初始化产生竞争条件：

```ts
React.useEffect(() => {
  if (!searchParams.get('keySerde')) { ... }
  if (!searchParams.get('valueSerde')) { ... }
  if (!searchParams.get('limit')) {
    searchParams.set('limit', MESSAGES_PER_PAGE);
  }
  // 新增：确保初始 URL 包含正确的默认方向和起始位置
  if (!searchParams.get('seekDirection')) {
    searchParams.set('seekDirection', SeekDirection.BACKWARD);
  }
  if (!searchParams.get('seekType')) {
    searchParams.set('seekType', SeekType.LATEST);
  }
  setSearchParams(searchParams);
}, [serdes]);
```

## 四、只读用户隐藏集群配置功能

**问题**：启用 `DYNAMIC_CONFIG_ENABLED=true` 后，只读用户在 Dashboard 页面仍可看到并使用"Configure new cluster"和"Configure"按钮。

**原因**：原有权限判断只检查用户是否拥有 `APPLICATIONCONFIG` 资源的任意权限，未区分 `VIEW` 和 `EDIT` 动作，导致只有查看权限的用户也会通过判断。

### 修改文件一：Dashboard 页面

**文件**：`kafka-ui-react-app/src/components/Dashboard/Dashboard.tsx`

```ts
// 修改前：只检查资源类型
return !!data?.userInfo?.permissions.some(
  (permission) => permission.resource === ResourceType.APPLICATIONCONFIG
);

// 修改后：同时检查必须包含 EDIT 动作
return !!data?.userInfo?.permissions.some(
  (permission) =>
    permission.resource === ResourceType.APPLICATIONCONFIG &&
    permission.actions.includes(Action.EDIT)
);
```

### 修改文件二：集群列表操作列

**文件**：`kafka-ui-react-app/src/components/Dashboard/ClusterTableActionsCell.tsx`

同上，将权限判断改为同时验证 `resource` 和 `Action.EDIT`，无权限时返回 `null` 隐藏按钮。

## 六、构建方式

在项目根目录执行以下命令，构建包含前端的完整 JAR：

```bash
./mvnw clean package -pl kafka-ui-api -am -Pprod -Ddocker.skip=true -DskipTests
```

构建产物：`kafka-ui-api/target/kafka-ui-api-0.0.1-SNAPSHOT.jar`

将此 JAR 替换到运行中的容器后重启即可生效：

```bash
docker cp kafka-ui-api/target/kafka-ui-api-0.0.1-SNAPSHOT.jar <容器ID>:/app/kafka-ui-api-0.0.1-SNAPSHOT.jar
docker restart <容器ID>
```
